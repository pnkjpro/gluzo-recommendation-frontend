Gluzo Documentation

Gluzo is a skincare recommendation platform that helps users discover products based on their skin profile. Users answer a short questionnaire, and the recommendation engine ranks products based on how well they match the user’s needs. Authenticated users receive more personalized recommendations because their purchase history is also considered.

1. Business Overview

What Problem Does Gluzo Solve?

Choosing skincare products can be confusing because every user has different:

  • Skin types
  • Skin concerns
  • Age groups
  • Budget preferences
  • Ingredient sensitivities

Gluzo simplifies this process by:

  1. Understanding the user’s profile through a questionnaire.
  2. Matching products against that profile.
  3. Ranking products with a confidence score.
  4. Providing explanations for every recommendation.

2. User Journeys

The platform supports two types of users:

Guest Users

Users who have not logged in.

Authenticated Users

Users who are registered and logged in.

3. Guest User Journey

This is the default experience for new visitors.

Step 1: User Visits Gluzo

User lands on:

  • Homepage
  • Product listing page
  • Recommendation flow

No account is required.

Step 2: User Completes Questionnaire

Questions include:

  • Skin Type
  • Skin Concerns
  • Age Range
  • Budget Preference
  • Allergies (optional)

Example:

  • Skin Type → Oily
  • Concerns → Acne, Dark Spots
  • Budget → Budget Friendly
  • Age → 18–24

Step 3: Responses Are Saved

Since the user is not logged in:

  • No user account exists yet.
  • System generates a unique session_id.
  • Questionnaire is stored against this session.

Example:

session_id = abc-123-xyz
user_id = null

This allows recommendations without registration.

Note: If the user submits the questionnaire again, the previous answers are completely replaced — each user (or session) only has one active questionnaire response at a time.

Step 4: Recommendations Are Generated

The recommendation engine:

  • Loads questionnaire answers.
  • Compares answers against all products.
  • Calculates confidence scores.
  • Returns best matching products.

Guest recommendations are based on:

✅ Skin Type

✅ Concerns

✅ Budget

✅ Age Range

❌ Purchase History (not available)

Step 5: User Browses Products

User can:

  • View recommendations
  • Browse product catalog
  • View product details

Without creating an account.

4. Authenticated User Journey

Authenticated users receive a more personalized experience.

Option A: Direct Registration

Registration Flow

  1. User enters:

    • Name
    • Email
    • Password

  2. OTP is sent to email.

  3. User verifies OTP.

  4. Account is created.

  5. User is automatically logged in.

Option B: Login

Login Flow

  1. User enters email and password.
  2. System validates credentials.
  3. Access token is issued.
  4. User session starts.

5. Guest-to-User Migration (Critical Flow)

This is one of the most important business flows in Gluzo.

Scenario

A guest user:

  1. Completes questionnaire.
  2. Receives recommendations.
  3. Decides to register later.

Without migration, all recommendation data would be lost.

What Gluzo Does

When user logs in or registers:

System checks:

Does a guest session exist?

If YES:

Guest questionnaire is migrated

Before:

user_id = null
session_id = abc-123

After:

user_id = 15
session_id = abc-123

Result:

✅ Questionnaire remains intact

✅ Recommendation history remains intact

✅ User continues exactly where they left off

Migration Happens During

Login

POST /api/users/login

OTP Registration

POST /api/users/otp-register

OTP Verification

POST /api/users/otp/verify

This ensures guest data is never lost.

Important: If the user already has existing questionnaire data (for example, they completed a quiz while logged in before), the system keeps the existing data and does not overwrite it with the guest session. The user’s own data takes priority.

7. Recommendation Engine

The recommendation engine is the heart of Gluzo.

Step 1: Load User Answers

Example:

Skin Type = Oily
Concern = Acne, Dark Spots
Budget = Budget
Age = 18-24

Step 2: Load Product Attributes

Each product contains attributes like:

Product:
- Oily Skin
- Acne Control
- Budget Friendly
- Salicylic Acid

Step 3: Compare User vs Product

The system checks:

Skin Type Match

User = Oily
Product = Oily

Match = 100%

Concern Match

User = Acne, Dark Spots
Product = Acne

Match = 50%

Budget Match

User = Budget
Product = Budget

Match = 100%

Age Match

User = 18-24
Product = 18-24

Match = 100%

Step 4: Calculate Confidence Score

Each attribute has a weight.

Attribute Weight
Skin Type 30
Concern 25
Budget 15
Age Range 10

Total questionnaire score:

80 Points

Step 5: Add Purchase History Score (Auth Users Only)

Authenticated users receive an additional:

20 Points

based on previous purchases.

Step 6: Guest Score Normalization

Since guest users don’t have purchase history, their scores would max out at 80/100. To ensure fairness, the system normalizes guest weights upward so their scores can still reach 100.

Attribute Auth Weight Guest Weight
Skin Type 30 37.5
Concern 25 31.25
Budget 15 18.75
Age Range 10 12.5
Purchase History 20 —
Total 100 100

Formula:

Multiplier = (80 + 20) / 80 = 1.25
Guest Weight = Original Weight × 1.25

This means a guest with a perfect questionnaire match scores 100, just like an authenticated user would.

Step 7: Partial Matching

When there is no exact match, the system checks for partial matches at 50% value.

Partial matching applies to three dimensions:

Dimension Partial Match Examples
Skin Type Combination ↔ Oily, Combination ↔ Dry
Budget Budget ↔ Mid-range, Mid-range ↔ Premium
Age Range 18-24 ↔ 25-34, 25-34 ↔ 35-44, etc.

Example:

User = Budget
Product = Mid-range
→ No exact match, but adjacent tier
→ Partial match = 50% of weight

This ensures products that are “close enough” still appear in recommendations, just ranked lower.

Step 8: Recommendation Reasons

Every recommendation includes human-readable explanations so users understand why a product was suggested.

Examples:

  • “Matches your skin type”
  • “Targets your skin concerns”
  • “Within your budget range”
  • “Suitable for your age range”
  • “You’ve purchased this before”
  • “Similar to products you’ve bought”

These reasons build user trust in the recommendations.

8. Purchase History Personalization

Only available for logged-in users.

The system checks previously purchased products.

Exact Product Purchased Before

Example:

User already bought Product A

Highest personalization boost.

Same Category

Example:

Purchased Cleanser
Recommended Cleanser

Strong boost.

Same Brand

Example:

Purchased Minimalist
Recommended Minimalist

Medium boost.

Similar Ingredients

Example:

Purchased product with Niacinamide
Recommended product with Niacinamide

Smaller boost.

Result

Returning customers receive increasingly personalized recommendations over time.

The system evaluates all four tiers but only uses the highest scoring one for each product. The boost percentages are:

Match Type Boost Points (out of 20)
Exact Product 100% 20
Same Category 70% 14
Same Brand 50% 10
Similar Ingredients 40% 8

9. Allergy Protection Layer

Before a product is scored, Gluzo checks allergies.

Example:

User Allergy

Retinol

Product Ingredients

Retinol
Hyaluronic Acid

Result:

Product is excluded

The product never appears in recommendations.

This acts as a safety filter before scoring begins.

10. Mock Purchase Flow

Since the platform focuses on recommendation quality rather than checkout functionality:

Current Approach

A “Purchase” button exists.

When clicked:

  1. No checkout flow is executed.
  2. A record is inserted into:
user_orders

This simulates a completed purchase.

Why?

It allows:

  • Purchase history generation
  • Recommendation personalization
  • Confidence score improvement

Without building:

  • Cart
  • Checkout
  • Payment gateway
  • Order management

Additionally, after a purchase is recorded, the system automatically clears the user’s recommendation cache so their next recommendation request reflects the new purchase history.
