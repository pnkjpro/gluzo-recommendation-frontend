Gluzo — API Documentation

A clean, human-friendly reference for every API endpoint in the Gluzo platform.
All endpoints are prefixed with /api. Authentication uses Bearer tokens (Laravel Sanctum).

Response Format

Every response follows the same envelope:

{
  "error": false,
  "status": "success",
  "message": "Human-readable message",
  "data": { ... },
  "success": true
}

On error: error: true, success: false, and an appropriate HTTP status code (400 / 403 / 404 / 422 / 500).

Rate Limiting

Route Group Limit Scope
Auth routes (/api/users/*) 10 requests/minute Per IP address
All other routes 60 requests/minute Per user (or IP if guest)

1. Authentication

1.1 Register (Email + Password)

POST /api/users/create

Creates a new user account. Sends a verification OTP to the user’s email before creating the account.

Field Type Required Rules
name string ✅ Max 255 characters
email string ✅ Must be unique
password string ✅ Min 6 characters, must include password_confirmation

What happens:

  1. A 6-digit OTP is sent to the provided email via Gmail SMTP.
  2. The user account is created in the database.
  3. Returns the new user object.

Response: 201 Created with user data.

1.2 Login (Email + Password)

POST /api/users/login

Authenticates a user and returns a Bearer token.

Field Type Required Rules
login string ✅ User’s email address
password string ✅ —
session_id string ❌ Guest session ID to migrate questionnaire data

What happens:

  1. Validates email and password.
  2. If email is not verified, a verification OTP is sent automatically.
  3. If session_id is provided, any guest questionnaire data is migrated to this user.
  4. Returns a Bearer token for future API calls.

Response: 200 OK with { user, token }.

1.3 Send OTP

POST /api/users/otp/send

Sends a 6-digit OTP code to the provided email address. The OTP is valid for 10 minutes.

Field Type Required Rules
email string ✅ Valid email
label string ✅ One of: reset_password, verify_email, login_otp

Important: For reset_password, the user must already exist (returns 400 otherwise).

Response: 200 OK with success message.

1.4 Verify OTP

POST /api/users/otp/verify

Verifies a previously sent OTP code and issues a Bearer token.

Field Type Required Rules
email string ✅ —
otp string ✅ 6-digit code
label string ✅ One of: verify_email, reset_password, login_otp
session_id string ❌ Guest session ID to migrate questionnaire data

What happens:

  1. Marks the OTP record as verified.
  2. For verify_email and login_otp: marks the user’s email as verified.
  3. If the user exists: issues a Bearer token and migrates any guest questionnaire data.
  4. If the user doesn’t exist: returns is_new: true — the frontend redirects to the registration page.

Response: 200 OK with { user, token, is_new }.

1.5 Complete Registration (OTP-first flow)

POST /api/users/otp-register

Registers a user who has already verified their email via OTP. Requires a recently verified OTP record (within 15 minutes).

Field Type Required Rules
name string ✅ —
email string ✅ Must be unique
password string ✅ Min 6 characters, must include password_confirmation
session_id string ❌ Guest session ID to migrate data

Response: 201 Created with { user, token }. Email is pre-verified.

1.6 Get Current User 🔒

GET /api/users/user — Requires authentication

Returns the currently authenticated user’s profile. Used by the frontend on page load to restore the session.

Response: 200 OK with user object.

1.7 Logout 🔒

POST /api/users/logout — Requires authentication

Revokes the current Bearer token, ending the session.

Response: 200 OK with success message.

1.8 Reset Password 🔒

POST /api/users/password/reset — Requires authentication

Field Type Required Rules
password string ✅ Min 6 characters, must include password_confirmation

Response: 200 OK with success message.

2. Questionnaire

2.1 Get Questions

GET /api/questionnaire/questions

Returns the list of quiz questions. Each question is dynamically built from the database’s attribute_types table. The response is cached for 1 hour for performance.

There are 5 questions:

Question Input Type Required Example Options
What is your skin type? Single select ✅ Oily, Dry, Combination, Normal, Sensitive
What are your primary skin concerns? Multi select ✅ Acne, Dark spots, Aging, Dryness, Large pores
What is your budget preference? Single select ✅ Budget, Mid-range, Premium
What is your age range? Single select ✅ 18-24, 25-34, 35-44, 45+
Any ingredient allergies? Multi select ❌ Retinol, Fragrance, Parabens, None

Response: 200 OK with array of question objects.

2.2 Submit Answers

POST /api/questionnaire

Saves the user’s quiz answers. If the user has previous answers, they are replaced (not appended).

Field Type Required Rules
answers object ✅ { "skin_type": ["oily"], "concern": ["acne", "dark_spots"], ... }
session_id string ❌ For guest users — tracks answers before login

What happens:

  1. Deletes any previous questionnaire response for this user (or session).
  2. Creates a new response record.
  3. Resolves each answer to its database attribute value IDs.
  4. Clears the user’s recommendation cache (so they get fresh results).

Response: 201 Created with { questionnaire_id, session_id }.

3. Recommendations

3.1 Get Recommendations

GET /api/recommendations

Returns a ranked list of skincare products scored against the user’s questionnaire answers. Results are cached for 10 minutes per user (or per guest session).

Param Type Required Rules
session_id string ❌ For guests who haven’t logged in
limit int ❌ 1–20, default: 10

How the system finds the user’s answers:

  1. If authenticated → uses the user’s latest questionnaire response.
  2. If guest → uses the provided session_id to find the response.
  3. If neither is found → returns 422: Please complete the questionnaire first.

Each recommendation includes:

  • Product details — name, brand, price, category, image
  • Confidence score — 0 to 100 (how well this product matches)
  • Reasons — human-readable explanations (“Matches your skin type”, “Within your budget range”)

Scoring breakdown:

  • Skin type match: up to 30 points
  • Concern match: up to 25 points
  • Budget match: up to 15 points
  • Age range match: up to 10 points
  • Purchase history (logged-in users only): up to 20 points

Products containing ingredients the user is allergic to are automatically excluded.

Response: 200 OK with { is_personalized, scoring_mode, recommendations: [...] }.

4. Products

4.1 List Products (with Infinite Scroll Pagination)

GET /api/products

Returns a paginated list of active products. The frontend implements infinite scrolling — as the user scrolls down, subsequent pages are fetched automatically and appended to the product grid.

Param Type Required Default Rules
category string ❌ — Filter by product category
per_page int ❌ 12 Number of products per page
page int ❌ 1 Page number

Products are sorted by category, then by name.

Response:

{
  "data": {
    "products": [
      {
        "id": 1,
        "name": "Salicylic Acid Face Wash",
        "brand": "Minimalist",
        "price": 349.00,
        "category": "Cleanser",
        "slug": "salicylic-acid-face-wash",
        "description": "...",
        "image_url": null
      }
    ],
    "pagination": {
      "current_page": 1,
      "per_page": 12,
      "total": 30,
      "last_page": 3
    }
  }
}

Frontend behavior:

  • Page loads → fetches page 1 (12 products)
  • User scrolls near the bottom → fetches page 2 (next 12 products), appends to the grid
  • Continues until all products are loaded
  • Shows “✓ All products loaded” when the last page is reached

4.2 Product Detail

GET /api/products/{id}

Returns a single product with all its attribute values (skin type tags, concern tags, ingredients, etc.).

Response: 200 OK with full product object and related attributes.

5. Orders 🔒

Both endpoints require authentication (Bearer token).

5.1 Record a Purchase

POST /api/orders/mock-purchase — Requires authentication

Records a mock purchase. This data is used by the recommendation engine to improve future suggestions (products in the same category or brand get a scoring boost).

Field Type Required Default Rules
product_id int ✅ — Must reference an existing product
quantity int ❌ 1 Range: 1–10

What happens:

  1. Creates an order record.
  2. Clears the user’s recommendation cache (so future recommendations reflect the purchase).

Response: 201 Created with { order_id, product, purchased_at }.

5.2 Purchase History

GET /api/orders — Requires authentication

Returns the user’s complete order history, sorted newest first, with full product details included.

Response: 200 OK with array of order objects.

Guest → User Session Migration

When a guest completes the questionnaire without being logged in, their answers are stored with a session_id (UUID). When they later log in or register:

  1. The system checks if the user already has questionnaire data.
  2. If no → the guest’s data is linked to their new user account.
  3. If yes → the existing user data is kept (takes priority).

This happens automatically during login, OTP verification, and registration — no extra API calls needed. The guest never loses their quiz answers.
